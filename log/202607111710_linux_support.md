# Linux support (Ubuntu 24.04, gcc/Ninja) without breaking the Windows build

Date: 2026-07-11
Commits: (uncommitted working tree ‚Äî see "Files changed")

## Goal

Make the (previously Windows/MSVC-only) build configure, compile, test, and
run on this Ubuntu 24.04 machine, and run **all** examples here ‚Äî while
keeping the existing MSVC/Windows path working byte-for-byte.

Machine: NVIDIA RTX 6000 Ada + NVIDIA `nvidia_icd.json`, `libvulkan-dev`
1.3.275 (system loader), X11 session (`DISPLAY=:0`), cmake 4.0.3, g++ 14.2,
Ninja 1.11, 56 cores. No LunarG SDK, no glslang, no `VULKAN_SDK` initially.

## What was done

All engine/renderer C++ was already portable (VulkanWrapper carries GCC/Clang
warning-suppression + Android branches; glad uses `gladLoadVulkanUserPtr` with
GLFW's proc loader, so no platform loader code; the core is headless via
`VkContext`, GLFW is only the optional live viewer). The work was entirely in
the **build glue** plus **one missing-include fix** masked by MSVC.

- **LunarG Vulkan SDK 1.4.350.1** installed from the Linux tarball into
  `~/vulkan/1.4.350.1` (no root; the tarball's prebuilt `x86_64/` tree ships
  the glslang static libs + headers + validation layers). Version matches the
  Windows preset's 1.4.350.0.
- **`cmake/FindVulkanSdk.cmake`** ‚Äî added a Linux/macOS discovery branch
  (globs `~/vulkan/*/x86_64`, `~/VulkanSDK/*/x86_64`, `/opt/vulkan/*/x86_64`;
  `VULKAN_SDK` env still wins). Now resolves include/lib subdir **casing**
  once into `DRESSI_VULKAN_SDK_INCLUDE` / `_LIB` (Windows `Include`/`Lib` vs
  Linux `include`/`lib`) so downstream targets are casing-agnostic. Windows
  discovery (`C:/VulkanSDK/*`) unchanged.
- **`CMakeLists.txt`**:
  - glslang linking split by platform. Windows keeps the exact original
    (`Include/glslang` + `<name>.lib`/`d.lib`). Linux adds **both** include
    roots (`include` for `<glslang/Public/...>`, `include/glslang` for
    `<SPIRV/...>` ‚Äî the Linux SDK flattens glslang's tree one level shallower
    than Windows) and links the static `lib*.a` inside
    `-Wl,--start-group ... --end-group` (the archives are mutually recursive:
    glslang ‚Üî MachineIndependent ‚Üî SPIRV).
  - `dressi_glad` include now uses `DRESSI_VULKAN_SDK_INCLUDE`.
  - `NOMINMAX` / `WIN32_LEAN_AND_MEAN` gated under `if(WIN32)`; added
    `Threads::Threads` + `${CMAKE_DL_LIBS}` on non-Windows (glslang/SPIRV-Tools
    archives pull in pthread + dlopen).
  - `GLFW_BUILD_WAYLAND OFF` / `GLFW_BUILD_X11 ON` on `UNIX AND NOT APPLE`:
    GLFW 3.4 otherwise demands `wayland-scanner` + libwayland/xkbcommon-dev.
    X11-only still runs under Wayland via XWayland. Overridable.
  - `CMAKE_CXX_SCAN_FOR_MODULES OFF`: the project uses no C++20 modules;
    Ninja+GCC would otherwise scan every TU with `-fmodules-ts` (pure
    overhead). Harmless on the VS generator.
- **`CMakePresets.json`** ‚Äî added `linux` configure preset (Ninja, gcc/g++,
  `CMAKE_BUILD_TYPE=Release`) + `linux-release` build/test presets. MSVC
  presets untouched.
- **`third_party/VulkanWrapper/include/vkw/vkw.h`** (submodule) ‚Äî added the
  standard includes it relied on MSVC to pull in transitively: `<memory>`
  (the actual blocker ‚Äî `std::shared_ptr` at vkw.h:92), plus `<array>`,
  `<string>`, `<tuple>`, `<vector>`. Idempotent/safe for Windows.

## How to build & run on Linux

```sh
cmake --preset linux                 # discovers ~/vulkan/<ver>/x86_64
cmake --build --preset linux-release
ctest --preset linux-release              # all tests
ctest --preset linux-release -LE gpu      # CPU-only
./build/examples/pbr_shading --frames=60  # headless orbit + PNGs
```

Binaries land at `build/tests/<name>` and `build/examples/<name>` (single-config,
no `Release/` subdir, no `.exe`).

## Key decisions

- **SDK over apt/FetchContent** (user choice): mirrors the Windows mental
  model (a `VULKAN_SDK` root with glslang + validation layers), so the CMake
  stays one shape with per-OS casing/lib-naming rather than two provisioning
  strategies.
- **System loader + SDK headers.** The SDK tarball ships no `libvulkan.so`;
  `find_package(Vulkan)` links the system loader (1.3.275) while headers come
  from the SDK (1.4.350). Safe because vkw uses the dynamic dispatcher
  (`VULKAN_HPP_DISPATCH_LOADER_DYNAMIC`) ‚Äî only `vkGetInstanceProcAddr` is
  needed from the loader; everything else resolves at runtime against the
  NVIDIA driver.
